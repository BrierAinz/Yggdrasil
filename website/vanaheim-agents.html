Vanaheim · AI Agents

The Vanir Pantheon

Vanaheim is the realm of magic and artificial intelligence. Here live the Vanir — a council of specialist agents, each with a distinct personality, LLM backend, and purpose. They share memory through Muninn, communicate across realms via Bifrost, and coordinate through a Council system.

Architecture

How Vanaheim Works

Each agent inherits from VanirAgent — a base class that defines the contract: execute, stream, is_available, health. They register in the VanirRegistry, load personas from personas.json, and persist memories via Muninn. The Bifrost Gateway bridges them to Asgard's Hermes-Lilith.

🧑‍💼 You Telegram / CLI
→
🤖 Telegram Bot Bots_Lilith_v5
→ HTTP →
🌉 Bifrost Gateway :8000
→
⚡ Hermes-Lilith Asgard
→ VanirAgent base class — Every agent implements execute(), stream(), is_available(), and health(). State machine: IDLE → BUSY → IDLE (or ERROR).
→ Persona system — Each agent has a unique system prompt loaded from personas.json. Personalities are distinct and persistent.
→ Muninn memory — Shared vector memory across agents. Context is injected per-query for continuity across sessions.
→ Bifrost Gateway — REST bridge connecting Vanaheim agents to Asgard's LilithEngine. JWT auth, streaming, and session management.
The Council

Agent Profiles

Five agents form the Vanir Pantheon. Each has a unique role, LLM backend, and personality. Together they form a council that can analyze, code, converse, classify, and create.

Odín — The All-Seeing Sage

The wise researcher of the Pantheon. Odín absorbed the creative capabilities of the former Lucifer agent, merging deep analysis with storytelling. He processes up to 262K tokens of context, making him the heavyweight for massive document analysis and architectural reviews.

  • Backend: Kimi (kimi-for-coding, 262K context)
  • Specialties: Deep analysis, research, creative writing, synthesis, architectural review
  • Intent detection: Switches between Analysis, Creative, and Default modes
  • Streaming: Yes
  • Memory: Muninn (10 context entries per query)
262K ctx streaming multi-mode Kimi

Eva — The Meticulous Analyst

The methodical one. Eva specializes in long-context analysis and documentation. Powered by Grok/xAI, she excels at parsing massive documents, synthesizing insights, and producing structured summaries. Her 128K context window makes her ideal for research-heavy tasks.

  • Backend: Grok/xAI (grok-4-fast-reasoning, 128K context)
  • Specialties: Analysis, documentation, insights, summarization, research
  • Streaming: Yes
  • Memory: Muninn (3 context entries per query)
128K ctx streaming research Grok

Crystal — The Public Face

The conversationalist. Crystal is the public-facing assistant designed for Discord and community interaction. She uses Kimi as her primary backend with automatic Ollama fallback — always available, always responsive. Her personality is warm, helpful, and approachable.

  • Backend: Kimi (primary) + Ollama (fallback, llama3.2)
  • Specialties: Conversation, general assistance, FAQ, help
  • Conversation history: Tracks last 5 messages per user
  • Streaming: Yes (Kimi), fallback to Ollama
  • Memory: Muninn (user + channel scoped)
8K ctx fallback Discord Kimi+Ollama

Adán — The Code Artisan

The builder. Adán runs entirely locally via Ollama (qwen2.5-coder:7b) — no API keys, no cloud, no latency. He generates code, writes tests, refactors, and debugs. The only agent with tool support, making him the hands-on craftsman of the Pantheon.

  • Backend: Ollama local (qwen2.5-coder:7b, 32K context)
  • Specialties: Code generation, refactoring, tests, debugging, code review
  • Tool support: Yes (the only agent with tools)
  • Streaming: Yes
  • Privacy: Runs 100% locally — no data leaves the machine
32K ctx local-first tools Ollama

Shalltear — The Tactical Classifier

The triage specialist. Shalltear runs on Venice AI (llama-3.3-70b) and excels at fast classification, natural language parsing, and intent detection. She is the gatekeeper — the one who decides which agent handles a task and how complex it is. Fast, decisive, tactical.

  • Backend: Venice AI (llama-3.3-70b, 32K context)
  • Specialties: Classification, parsing, triage, intent detection
  • Streaming: Yes (rarely needed — single-shot responses)
  • Memory: Muninn (contextual classification memory)
  • Role: Router — classifies task → delegates to the right agent
32K ctx triage fast Venice AI

Infrastructure

Council System & Shared Services

Beyond individual agents, Vanaheim provides shared infrastructure that all agents use: a unified framework, memory system, API gateway, and persona management.

🏛️
Council System Agents coordinate through a council mechanism. Shalltear classifies the task intent, then delegates to the appropriate specialist. Complex tasks can be routed to multiple agents for parallel processing.
🧠
Muninn Memory Client Every agent persists conversations and results through muninn_client — a shared memory layer. Context is retrieved per-query, giving agents continuity across sessions without bloating prompts.
🎭
Persona Loader Each agent loads a unique system prompt from personas.json. The PersonaLoader caches prompts and supports inheritance — common instructions are shared, agent-specific directives override.
🌉
Bifrost Gateway A FastAPI REST bridge (port 8000) connecting Vanaheim to Asgard's LilithEngine. JWT authentication, streaming support, and session management. The Telegram Bot routes all messages through this gateway.
🤖
vanaheim-framework A Python bot framework with BaseBot abstract class, health checks, and structured logging. Provides the foundation for all bots in Vanaheim — from the main Telegram bot to specialized agents.
📱
Telegram Bot (Bots_Lilith_v5) The remote control interface for the entire ecosystem. Routes messages through LilithBridge to the Gateway, with structured logging, heartbeat monitoring, and retry management.
📋
VanirRegistry A JSON-based agent registry (vanir_registry.json) that tracks agent state, configuration, capabilities, and health. Every agent self-registers on startup and reports heartbeat updates.
⚡
Agent Lifecycle State machine per agent: IDLE → BUSY → IDLE (or ERROR). Execution timing is tracked. Each agent reports health, availability, and current task through the registry API.

Cross-Realm

Bifrost Protocol

Vanaheim does not exist in isolation. The Bifrost bridge connects it to Asgard's core infrastructure, allowing Telegram messages to flow through the Gateway into Lilith and back. Agents in Vanaheim can leverage Asgard's memory, scheduler, and tools.

→ Telegram → Bots_Lilith_v5 → LilithBridge → Bifrost Gateway → LilithEngine — The full message pipeline from user to AI.
→ Vanaheim agents can call Asgard's memory system directly through the Gateway REST API.
→ Muninn (memory client) stores agent-specific context — each agent has its own memory namespace.
→ When agents mature, they migrate to Midgard (personal apps) or Svartalfheim (knowledge/docs) following the Yggdrasil lifecycle rules.
